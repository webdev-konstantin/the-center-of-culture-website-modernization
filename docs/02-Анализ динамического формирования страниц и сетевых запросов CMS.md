# День 2 — Анализ динамического формирования страниц и сетевых запросов CMS

**Дата:** 2026-07-01

## Цель дня

Продолжить исследование механизма формирования раздела «Библиотека и студии», определить источник данных для его элементов и установить, каким образом административная панель, серверная часть и база данных взаимодействуют между собой.

## Выполненные задачи

### 1. Продолжение reverse engineering самописной CMS

Работа второго дня была сосредоточена на восстановлении внутренней логики CMS и поиске источника данных для раздела «Библиотека и студии».

На начало исследования было известно, что:

- раздел отображается на главной странице сайта;
- его элементы доступны в визуальном редакторе CMS;
- карточки содержат изображения, подписи и ссылки;
- содержимое раздела не обнаруживается в обычном HTML-поле страницы;
- кнопка просмотра исходного кода в редакторе не показывает ожидаемую разметку;
- система использует собственные виджеты;
- CMS не имеет доступной технической документации.

Поэтому исследование продолжалось методом последовательной проверки всех возможных уровней системы:

- административного интерфейса;
- визуального редактора;
- PHP-файлов;
- JavaScript-компонентов;
- сетевых запросов;
- базы данных MySQL;
- связей между таблицами;
- параметров CMS-виджетов.

### 2. Анализ файловой структуры проекта

Через файловый менеджер хостинга была продолжена проверка структуры сайта.

Особое внимание уделялось:

- PHP-файлам административной части;
- серверным обработчикам запросов;
- каталогам с JavaScript;
- каталогу `WEB-INF`;
- файлам, связанным с выборкой данных;
- возможным конфигурационным файлам базы данных.

В ходе анализа были обнаружены PHP-файлы, выполняющие роль универсальных обработчиков запросов.

Одним из ключевых файлов оказался:

```text
select.php
```

Первоначально предполагалось, что этот файл может содержать прямое подключение к базе данных или собственную логику формирования страниц.

Однако анализ показал, что `select.php` является универсальным SQL-интерфейсом, который принимает параметры запроса и выполняет выборку данных из разрешённых таблиц.

Общая логика запроса имеет вид:

```sql
SELECT $_POST['fields']
FROM $_POST['tables']
WHERE $_POST['condition']
```

Это означало, что сам файл `select.php`:

- не хранит содержимое страниц;
- не содержит разметку раздела «Библиотека и студии»;
- не определяет структуру главной страницы;
- выполняет роль промежуточного обработчика между интерфейсом CMS и MySQL.

### 3. Исследование серверных запросов CMS

В браузерных инструментах разработчика были изучены сетевые обращения административной панели.

В запросах обнаружились обращения к различным таблицам базы данных, включая:

```text
tables=content
```

и:

```text
tables=fragments
```

Это подтвердило, что административная панель не хранит редактируемые страницы в статическом виде.

Вместо этого интерфейс CMS:

1. отправляет AJAX-запрос;
2. передаёт имя необходимой таблицы;
3. получает данные от PHP-обработчика;
4. преобразует ответ в объекты интерфейса;
5. динамически формирует визуальный редактор.

Таким образом, административная панель работает как клиентское приложение поверх базы данных.

### 4. Анализ клиентской части и JavaScript

В структуре проекта были обнаружены многочисленные файлы:

```text
*.cache.js
```

а также файл:

```text
ruscenter.nocache.js
```

Подобная структура характерна для приложений, собранных с использованием Google Web Toolkit.

Это позволило предположить, что административная панель CMS является GWT-приложением, в котором:

- основной интерфейс написан на Java;
- Java-код скомпилирован в JavaScript;
- большая часть логики находится в сгенерированных `.cache.js`-файлах;
- PHP используется преимущественно для обмена данными с сервером;
- страницы редактора создаются динамически на стороне браузера.

Это объяснило, почему в проекте не удалось сразу найти обычные PHP-шаблоны, полностью соответствующие экрану административной панели.

### 5. Проверка браузерной консоли

Для исключения ошибок JavaScript была открыта вкладка `Console` в Chrome DevTools.

В консоли были обнаружены предупреждения Mixed Content.

Пример:

```text
Mixed Content:
The page was loaded over HTTPS,
but requested an insecure element over HTTP.
```

Некоторые изображения запрашивались по адресам вида:

```text
http://ruscenter.hu/uploads/...
```

при том что сама административная панель была открыта по HTTPS.

Chrome автоматически заменял такие запросы на HTTPS.

Эти предупреждения были зафиксированы как технический долг, но не являлись причиной исчезновения раздела.

Также в консоли присутствовала ошибка вида:

```text
Uncaught (in promise)
Error: A listener indicated an asynchronous response
by returning true, but the message channel closed
before a response was received
```

По характеру сообщения было установлено, что ошибка, вероятнее всего, создавалась одним из браузерных расширений и не относилась непосредственно к CMS.

Критических ошибок вида:

```text
CKEDITOR is not defined
```

```text
TypeError
```

```text
Cannot read property
```

в консоли обнаружено не было.

Следовательно, гипотеза о полном отказе JavaScript-редактора не подтвердилась.

Команда:

```text

```

указывала, что содержимое самой страницы необходимо искать не в `coursesandstudios`, а в другой таблице.

Была проверена таблица:

```text
content
```

Поиск выполнялся по значению:

```text

```

или SQL-запросом:

```sql
SELECT *
FROM content
WHERE hash = 'hash7';
```

Следовательно, страница «Библиотека»:

### 13. Исследование виджета «Панель ярлыков»

При открытии свойств виджета было обнаружено окно:

```text

```


Именно поэтому виджет существовал в структуре страницы, но не знал:



### 16. Разделение ответственности между таблицами

В процессе исследования было установлено, что данные распределены между несколькими уровнями.

#### JavaScript/GWT-интерфейс

Выполняет:

- формирование административной панели;
- отображение визуального редактора;
- обработку действий пользователя;

### 17. Проверенные гипотезы

Во время исследования было последовательно проверено несколько возможных причин исчезновения раздела.

#### Гипотеза 1 — отсутствует PHP-файл подключения к базе данных

Не подтвердилась.

CMS использует серверную инфраструктуру и универсальные обработчики, а подключение может быть реализовано вне исследованных PHP-файлов или внутри серверного окружения приложения.

#### Гипотеза 2 — содержимое находится в `select.php`

Не подтвердилась.

`select.php` выполняет только универсальные SQL-запросы.

#### Гипотеза 3 — данные находятся в таблице `fragments`

Не подтвердилась как основная причина.

Таблица участвует в работе CMS, но карточки раздела были обнаружены в `coursesandstudios`.

#### Гипотеза 4 — страница «Библиотека» отсутствует

Не подтвердилась.

Страница `hash7` существует и опубликована.

#### Гипотеза 5 — редактор не работает из-за JavaScript-ошибки

Не подтвердилась.

Критических ошибок редактора в консоли не обнаружено.

#### Гипотеза 6 — проблема вызвана Mixed Content

#### Гипотеза 7 — повреждены данные таблицы `coursesandstudios`

Не подтвердилась.

Все необходимые записи присутствовали.

#### Гипотеза 8 — отсутствуют параметры виджета

Подтвердилась.

## Ключевые технические находки

### 1. CMS использует динамическое формирование страниц

Главная страница не хранится в виде единого статического HTML-документа.

Она собирается из:

- текстовых блоков;
- изображений;
- баннеров;
- специализированных виджетов;
- записей базы данных.

### 2. Административная панель является клиентским приложением

Интерфейс CMS работает поверх JavaScript/GWT и получает данные через AJAX.

### 3. PHP выполняет роль серверного посредника

PHP-файлы принимают параметры от административной панели и обращаются к MySQL.

### 4. Страницы и карточки хранятся отдельно

Таблица `content` хранит страницы, а `coursesandstudios` — карточки раздела.

### 5. Навигация реализована через hash

Пример:

```text
!page=hash7
```

Значение `hash7` соответствует странице «Библиотека» в таблице `content`.

### 6. Виджеты требуют собственных параметров

Для «Панели ярлыков» необходимы:

```text
columns
dbtable
locale
```

Без этих параметров CMS не знает, откуда загружать элементы.

### 7. Параметры виджета доступны через визуальный редактор

Они не были обнаружены:

- в обычном HTML;
- в поле `content.html`;
- непосредственно в карточках;
- в открытых PHP-шаблонах.

Получить их удалось только через свойства виджета в редакторе CMS.

## Основной результат дня

Удалось полностью восстановить механизм формирования раздела «Библиотека и студии».

Было установлено, что раздел формируется виджетом «Панель ярлыков», который получает элементы из таблицы:

```text
coursesandstudios
```

Карточки этого раздела содержат ссылки на страницы, расположенные в таблице:

```text
content
```

```text
!page=hash7
```

```text
columns = 4

locale = ru
```

После восстановления этих значений:

- виджет снова получил доступ к таблице;

- ссылки на страницы начали работать;
- раздел «Библиотека и студии» снова появился в редакторе CMS.
