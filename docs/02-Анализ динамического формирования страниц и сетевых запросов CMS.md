# Выполненные задачи

## 1. Анализ серверной и клиентской частей CMS

Через файловый менеджер хостинга были исследованы:

- PHP-файлы;
- JavaScript-каталоги;
- административная часть;
- каталог `WEB-INF`;
- обработчики запросов;
- возможные файлы конфигурации базы данных.

Одним из ключевых файлов оказался:

```text
select.php
```

Анализ показал, что этот файл не хранит содержимое страниц, а выполняет запросы к базе данных на основании переданных параметров.

Общая логика:

```sql
SELECT $_POST['fields']
FROM $_POST['tables']
WHERE $_POST['condition']
```

`select.php` выполняет роль посредника:

```text
Интерфейс CMS
        ↓
POST-запрос
        ↓
select.php
        ↓
MySQL
        ↓
Ответ интерфейсу
```

## 2. Анализ сетевых запросов

Через Chrome DevTools были изучены обращения административной панели к серверу.

В запросах присутствовали параметры:

```text
tables = content
```

и:

```text
tables = fragments
```

Это подтвердило, что административная панель не хранит страницы как готовый статический HTML.

Вместо этого CMS:

1. отправляет AJAX-запрос;
2. указывает таблицу;
3. получает данные от PHP;
4. преобразует ответ в элементы интерфейса;
5. динамически формирует визуальный редактор.

## 3. Анализ JavaScript-структуры

В проекте были обнаружены файлы:

```text
*.cache.js
```

и:

```text
ruscenter.nocache.js
```

Такая структура соответствует приложению, собранному с использованием Google Web Toolkit.

Это объясняет, почему логика административной панели не находилась в обычных PHP-шаблонах.

Архитектура клиентской части выглядит следующим образом:

```text
GWT-приложение
        ↓
Скомпилированный JavaScript
        ↓
AJAX-запросы
        ↓
PHP
        ↓
MySQL
```

В таблице:

```text

```

были найдены записи, соответствующие карточкам раздела.

| `id` | `text` | `command` |
|---:|---|---|
| 7 | Студии | пусто |

| 20 | Логопед | ссылка |

Основные поля таблицы:

| Поле | Назначение |
|---|---|
| `locale` | языковая версия |
| `seqno` | порядок отображения |

Пример записи «Библиотека»:

```text
id = 22

command = !page=hash7
icon = uploads/coursesandstudios/img1723.jpg
locale = ru
iconheight = 120
```

Карточка не содержит полную страницу. Она хранит ссылку:

```text
!page=hash7
```

## 7. Связь карточки со страницей

```text

```


Были найдены две языковые версии:

- русская;
- венгерская.

Для русской версии подтверждены значения:

```text

```

Запись также содержала:

- HTML-содержимое;
- изображение;
- параметры публикации;
- языковую версию.

Связь карточки и страницы:

```text
coursesandstudios.command
        ↓
!page=hash7
        ↓
content.hash
        ↓
hash7
        ↓
Страница «Библиотека»
```

## 8. Роль таблицы `content`

Таблица `content` является основным хранилищем страниц CMS.

Ключевые поля:

```text
id
classid
locale
hash
published
header
subheader
thumbnail
html
```

Общая структура записи:

```text
hash
        ↓
Идентификатор материала
        ↓
locale
        ↓
Языковая версия
        ↓
header
        ↓
Заголовок
        ↓
html
        ↓
Основное содержимое
```

Для главной страницы используется:

```text
hash = main
```

Для страницы библиотеки:

```text
hash = hash7
```

Карточки и страницы хранятся отдельно:

```text
coursesandstudios
        ↓
Карточки и оформление

content
        ↓
Страницы и основное содержимое
```

## 12. Параметры виджета

В свойствах «Панели ярлыков» были обнаружены три параметра:

```text
columns
dbtable
locale
```

На момент исследования они были пустыми.

Из-за этого виджет не знал:

- сколько колонок выводить;
- из какой таблицы загружать карточки;
- какую языковую версию использовать.

Для русской версии были восстановлены значения:

```text
columns = 4
dbtable = coursesandstudios
locale = ru
```

Назначение параметров:

| Параметр | Назначение |
|---|---|
| `columns` | количество колонок |
| `dbtable` | источник данных |
| `locale` | язык карточек |

Для венгерской версии требуется:

```text
columns = 4
dbtable = coursesandstudios
locale = hu
```
