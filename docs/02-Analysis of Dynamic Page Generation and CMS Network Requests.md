### 1. Continued reverse engineering of the custom CMS

The second day focused on reconstructing the internal logic of the CMS and locating the data source used by the **ÔÇťLibrary and StudiosÔÇŁ** section.

At the beginning of the investigation, the following facts were known:

- the section was displayed on the website homepage;

- the section content could not be found in the standard HTML field of the page;
- the source-code view button in the editor did not show the expected markup;
- the system used its own custom widgets;
- no technical documentation for the CMS was available.

The investigation therefore continued by systematically examining every possible layer of the system:

- the administrative interface;
- the visual editor;

### 2. Analysis of the project file structure

The website structure was examined further through the hosting file manager.

Particular attention was paid to:

- PHP files used by the administrative interface;
- server-side request handlers;
- JavaScript directories;
- the `WEB-INF` directory;
- files related to data retrieval;
- possible database configuration files.

Several PHP files acting as universal request handlers were identified during the analysis.

One of the key files was:

```text
select.php
```

Initially, it was assumed that this file might contain a direct database connection or its own page-generation logic.

However, the analysis showed that `select.php` acts as a universal SQL interface that accepts request parameters and retrieves data from permitted tables.

The general query logic has the following form:

```sql
SELECT $_POST['fields']
FROM $_POST['tables']
WHERE $_POST['condition']
```

This meant that `select.php` itself:

- does not store page content;
- does not contain the markup of the ÔÇťLibrary and StudiosÔÇŁ section;
- does not define the homepage structure;
- acts as an intermediary handler between the CMS interface and MySQL.

---

### 3. Investigation of CMS server requests

The network requests generated by the administrative interface were examined using browser developer tools.

The requests included calls to different database tables, including:

```text
tables=content
```

and:

```text
tables=fragments
```

This confirmed that the administrative interface does not store editable pages in a static form.

Instead, the CMS interface:

1. sends an AJAX request;
2. passes the name of the required table;
3. receives data from the PHP handler;
4. converts the response into interface objects;
5. dynamically builds the visual editor.

The administrative interface therefore operates as a client-side application on top of the database.

---

### 4. Analysis of the client-side application and JavaScript

The project structure contained numerous files matching the following pattern:

```text
*.cache.js
```

as well as the file:

```text
ruscenter.nocache.js
```

This type of structure is characteristic of applications built with Google Web Toolkit.

This suggested that the CMS administrative interface is a GWT application in which:

- the main interface was written in Java;
- the Java code was compiled into JavaScript;
- much of the application logic is stored in generated `.cache.js` files;
- PHP is used primarily for server-side data exchange;
- editor pages are created dynamically in the browser.

This explained why ordinary PHP templates corresponding directly to the administrative screens could not be found immediately.

- construction of the administrative panel;

## Key technical findings

### 1. The CMS uses dynamic page generation

The homepage is not stored as a single static HTML document.

It is assembled from:

- text blocks;
- images;
- banners;
- specialized widgets;
- database records.

### 2. The administrative panel is a client-side application

The CMS interface runs on JavaScript/GWT and retrieves data through AJAX.

### 3. PHP acts as the server-side intermediary

PHP files receive parameters from the administrative interface and query MySQL.

### 4. Pages and cards are stored separately

The `content` table stores pages, while `coursesandstudios` stores the section cards.

### 5. Navigation is implemented through hashes

Example:

```text
!page=hash7
```

The `hash7` value corresponds to the ÔÇťLibraryÔÇŁ page in the `content` table.

### 6. Widgets require their own parameters

The **ÔÇťShortcut PanelÔÇŁ** requires:

```text
columns
dbtable
locale
```

Without these parameters, the CMS does not know where to load the elements from.

### 7. Widget parameters are available through the visual editor

They were not found:

- in ordinary HTML;
- in the `content.html` field;
- directly inside the card records;
- in the examined PHP templates.

They could only be accessed through the widget properties in the CMS editor.

## Main result of the day

The generation mechanism of the **ÔÇťLibrary and StudiosÔÇŁ** section was fully reconstructed.

It was established that the section is generated by the **ÔÇťShortcut PanelÔÇŁ** widget, which retrieves its elements from the following table:
