## Цель дня

Основной целью третьего дня стало определение возможности интеграции экспортированной главной страницы Tilda с существующей CMS **без изменения привычного процесса публикации материалов**.

Вместо ручного переноса новостей, событий и изображений было принято решение использовать существующую базу данных MySQL как единый источник данных.

Целевая архитектура выглядела следующим образом:

```text
Сотрудник публикует материал через существующую CMS
        ↓
CMS сохраняет запись в MySQL
        ↓
Новая главная страница получает последние записи
        ↓
Материал автоматически появляется на сайте
        ↓
Дизайн Tilda остаётся неизменным
```

В ходе исследования требовалось ответить на четыре основных вопроса:

1. Какие элементы экспортированной страницы уже работают самостоятельно?
2. Почему несколько информационных разделов отображают сообщение `Feed not found`?
3. Где именно существующая CMS хранит новости и события?
4. Каким образом текущий сайт получает и отображает опубликованные материалы?

## 5. Выбор стратегии интеграции

После анализа экспортированной страницы и существующей CMS было принято решение отказаться от использования оригинального механизма Tilda Feed.

Вместо него планируется использовать существующую CMS как источник данных.

Текущая схема выглядит так:

```text
Tilda Feed
        ↓
Публикации
        ↓
Карточки
```

Целевая схема модернизации:

```text
Существующая CMS
        ↓
MySQL
        ↓
PHP
        ↓
JSON
        ↓
JavaScript
        ↓
Карточки внутри дизайна Tilda
```

Такой подход позволяет сохранить:

- существующую административную панель;
- привычный процесс публикации материалов;
- базу данных MySQL;
- существующие изображения;
- поддержку нескольких языков;
- экспортированный дизайн Tilda.

Самое главное — сотрудники смогут продолжать работать в уже знакомой CMS, а новая главная страница будет автоматически получать опубликованные материалы без какого-либо ручного переноса.

## 6. Исследование структуры базы данных

Следующим этапом стало изучение того, каким образом существующая CMS хранит новости, мероприятия и страницы сайта.

Основной интерес представляла таблица:

```text
content
```

Именно она содержит материалы, отображаемые на сайте.

Во время исследования были определены основные поля таблицы.

| Поле | Назначение |
|---|---|
| `id` | уникальный идентификатор записи |
| `locale` | язык материала |
| `hash` | логический идентификатор страницы |
| `published` | статус публикации |
| `categ` | категория материала |
| `important` | приоритет отображения |
| `starttime` | дата публикации |
| `endtime` | дата окончания |
| `header` | заголовок |
| `subheader` | краткое описание |
| `thumbnail` | путь к изображению |
| `html` | основное содержимое |

Пример записи:

```text
id = 1560
locale = ru
hash = hash1560
published = 1
categ = 2
important = 9
starttime = 2024-07-04 02:00:00
endtime = 2024-10-10
thumbnail = uploads/content/hash1560/img2820.jpg
```

Связь записи с изображением выглядит следующим образом:

```text
Запись в content
        ↓
hash1560
        ↓
thumbnail
        ↓
uploads/content/hash1560/img2820.jpg
```

Это подтвердило, что CMS уже хранит всю информацию, необходимую для формирования карточек новостей:

- заголовок;
- краткое описание;
- даты;
- язык;
- категорию;
- приоритет;
- путь к изображению;
- полный текст.

## 7. Языковые версии материалов

Было установлено, что русская и венгерская версии материалов хранятся как отдельные записи.

Например:

```text
id 15 | locale = ru | hash = hash15
id 16 | locale = hu | hash = hash15
```

Обе записи относятся к одному материалу.

Структура выглядит так:

```text
hash15
├── locale = ru
└── locale = hu
```

Таким образом:

- `id` определяет конкретную запись;
- `hash` объединяет разные языковые версии;
- `locale` определяет язык отображения.

В дальнейшем это позволит переключать язык простым условием SQL:

```sql
WHERE locale = 'ru'
```

или

```sql
WHERE locale = 'hu'
```

## 8. Исследование SQL-запросов

Во время работы использовались только безопасные запросы:

```text
SELECT
```

Они не изменяли содержимое базы данных.

### Просмотр последних материалов

```sql
SELECT
    id,
    locale,
    published,
    categ,
    important,
    starttime,
    endtime,
    header,
    thumbnail
FROM content
WHERE locale = 'ru'
ORDER BY id DESC
LIMIT 30;
```

Запрос показал, что записи различаются:

- категорией;
- статусом публикации;
- приоритетом;
- датами;
- изображениями.

### Подсчёт материалов по категориям

```sql
SELECT
    categ,
    COUNT(*) AS total
FROM content
WHERE locale = 'ru'
GROUP BY categ
ORDER BY categ;
```

Результат:

| Категория | Количество |
|---:|---:|
| `0` | 54 |
| `1` | 321 |
| `2` | 371 |
| `3` | 7 |

Таким образом удалось подтвердить наличие четырёх основных категорий материалов.

### Исследование категории 3

```sql
SELECT
    id,
    header,
    starttime,
    thumbnail
FROM content
WHERE locale = 'ru'
  AND categ = 3
ORDER BY id DESC
```

Полученные записи были посвящены:

- международным мероприятиям;
- образовательным программам;

## 9. Логика публикации материалов

Следующим этапом стало изучение принципов отображения материалов на сайте.

Поле публикации использует два значения:

```text
published = 0
published = 1
```

Рабочие запросы сайта используют только опубликованные записи:

```sql
published = 1
```

Порядок отображения определяется следующим правилом:

```sql
ORDER BY important DESC,
         starttime DESC
```

Это означает:

1. сначала отображаются наиболее важные материалы;
2. затем выполняется сортировка по дате.

Например:

```text
important = 9
```

будет показан раньше, чем

```text
important = 4
```

Для формирования списка публикаций используются поля:

```text
published
important
starttime
endtime
```

Также были обнаружены старые записи со значениями:

```text
0000-00-00
0000-00-00 00:00:00
```

При создании нового API такие значения рекомендуется преобразовывать в:

```text
null
```

## 12. Реальный SQL-запрос сайта

Удалось восстановить одно из условий выборки материалов.

Оно имеет вид:

```text

        ↓

        ↓

5 последних записей
```

Именно этот запрос стал одним из главных результатов исследования.

## 13. Формат ответа сервера

Во вкладке **Response** было установлено, что сервер возвращает не готовую HTML-страницу, а структурированные данные.

Ответ может содержать:

```html
<p>
<strong>
<a>
<br>
```

Русские символы иногда передаются как Unicode:

```text
\u0434
\u0430
\u043d
\u043d
```

После получения ответа JavaScript выполняет следующие действия:

```text
Получение ответа
        ↓
Преобразование текста
        ↓
Обработка HTML
        ↓
Создание карточек
        ↓
Вывод материалов на страницу
```

Таким образом:

```text
select.php
```

возвращает только данные, а отображением занимается браузер.

## 15. Планируемая архитектура интеграции

Проведённое исследование показало, что экспортированная страница Tilda может использоваться как новый пользовательский интерфейс, а существующая CMS продолжит отвечать за хранение и публикацию материалов.

Планируемая архитектура выглядит следующим образом:

```text
Существующая CMS
        ↓
MySQL
        ↓
PHP API (только чтение)
        ↓
JSON
        ↓
JavaScript
        ↓
Блоки Tilda
```

В такой схеме:

- CMS продолжает использоваться для публикации материалов;
- MySQL остаётся единственным источником данных;
- новая главная страница отвечает только за отображение информации;
- сотрудники продолжают работать в привычной административной панели.

Переносить существующие новости и мероприятия вручную не потребуется.

## 16. Предлагаемый PHP API

Вместо передачи SQL-запросов из браузера рекомендуется создать отдельный API только для чтения данных.

Например:

```text
/api/home-feed.php
```

Основные задачи обработчика:

- определить требуемый раздел;
- определить язык;
- проверить количество запрашиваемых записей;
- выполнить заранее подготовленный SQL-запрос;
- сформировать ссылки на изображения;
- обработать даты;
- вернуть результат в формате JSON.

Пример запроса:

```text
/api/home-feed.php?type=news&locale=ru&limit=4
```

Пример ответа:

```json
{
  "status": "ok",
  "items": [
    {
      "id": 1560,
      "hash": "hash1560",
      "header": "Заголовок новости",
      "subheader": "Краткое описание",
      "thumbnail": "/uploads/content/hash1560/img2820.jpg",
      "starttime": "2024-07-04 02:00:00"
    }
  ]
}
```

Такой подход позволяет полностью оставить SQL-логику на стороне сервера и передавать браузеру только необходимые данные.

## 17. Интеграция с JavaScript

После создания API новая главная страница сможет получать материалы через JSON вместо использования Tilda Feed.

Последовательность работы будет следующей:

```text
JavaScript
        ↓
Запрос JSON
        ↓
Получение записей
        ↓
Создание карточек
        ↓
Вывод карточек в существующие блоки Tilda
```

Пример запроса:

```javascript
fetch('/api/home-feed.php?type=news&locale=ru&limit=4')
    .then(response => response.json())
    .then(data => {
        console.log(data.items);
    });
```

Каждая запись уже содержит всю информацию, необходимую для формирования карточки:

- заголовок;
- краткое описание;
- дату публикации;
- изображение;
- идентификатор страницы.

# Основной результат дня

Удалось полностью восстановить механизм получения материалов существующим сайтом.

Подтверждено, что:

- экспортированную страницу Tilda можно использовать без существенных изменений;
- только три информационных блока требуют нового источника данных;
- все материалы уже находятся в MySQL;
- существующая CMS получает их через `select.php`;
- JavaScript преобразует полученные данные в карточки страниц;
- существующую административную панель можно сохранить без изменений.

```text
Сотрудник
        ↓

JSON
        ↓
JavaScript
        ↓
Новая главная страница
```

# Практическое значение

Модернизация сайта может быть выполнена без замены существующей CMS.

Рабочий процесс останется прежним:

```text
Сотрудник публикует материал
        ↓
CMS сохраняет запись
        ↓
MySQL обновляется
        ↓
Главная страница получает новые данные
        ↓
Материал автоматически появляется на сайте
```

Это позволяет сохранить:

- существующую CMS;
- текущую базу данных;
- изображения;
- русскую и венгерскую версии;
- привычный процесс работы сотрудников;
- адаптивную вёрстку нового дизайна.
